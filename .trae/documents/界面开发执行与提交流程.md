# 界面开发执行与提交流程

本文档用于固定《我的高考志愿模拟器》后续界面开发方式，作为默认执行规范。

## 一、总原则

1. 先保证功能链路能跑通，再补充正式美术。
2. 每次只推进一个界面，不跨多个界面同时深入开发。
3. 当前界面经过你测试确认没有问题后，才进入该界面的正式美术替换和细化。
4. 当前界面完成并确认后，先提交到 Git 存档，再开始下一个界面。

## 二、单个界面的标准执行步骤

### 第 1 步：先做临时美术功能界面

目标：
- 快速搭出可运行、可点击、可验证流程的临时界面。
- 临时界面允许使用简化布局、占位图、运行时 UI、简单颜色块和基础按钮样式。

要求：
- 优先保证按钮、输入、跳转、状态写入、返回逻辑正确。
- 暂不追求最终视觉，只要结构清晰、能测功能即可。
- 做完后提供给你测试。

### 第 2 步：等待你测试确认

目标：
- 由你确认当前界面功能、流程、文案和交互是否正确。

要求：
- 如果你提出修改意见，先继续调整当前界面，不进入下一个界面。
- 在你明确表示“没问题”“可以继续”之前，不默认切到后续界面开发。

### 第 3 步：按既定风格生成正式美术并替换

目标：
- 在当前界面功能确认无误后，再生成与项目统一风格接近的正式美术素材并替换临时素材。

要求：
- 你提供的图片仅作为风格参考，不要求你每次都提供素材原图。
- 后续默认由我按照已确认风格自行生成所需美术素材，并接入界面。
- 如果某个角色、道具、场景需要严格对应指定形象，再由你补充原图或额外说明。
- 正式美术替换时，不能破坏已经确认通过的功能逻辑。

### 第 4 步：完成当前界面后提交 Git

目标：
- 把“功能确认 + 美术替换完成”的当前界面保存成可回退的稳定里程碑。

要求：
- 每次完成一个界面并确认无误后，立即执行一次 Git 提交。
- 提交完成后再继续下一个界面，方便你在家里接着开发或回看历史版本。

## 三、默认协作方式

后续默认按以下节奏执行：

1. 我先实现下一个界面的临时美术功能版。
2. 你测试当前界面并反馈问题。
3. 我修到通过后，再生成并替换正式美术素材。
4. 我完成最终检查。
5. 我提交 Git 存档。
6. 然后开始下一个界面。

## 四、美术执行约定

1. 后续美术风格默认参考你已经确认过的界面风格。
2. 人物相关界面默认沿用已确认的人物表现方式和统一配色方向。
3. 如果只是“风格参考图”，默认由我自行生成同风格素材，不再反复等你单独出图。
4. 如果是“必须指定的形象或元素”，你提前说明，我按指定内容生成或替换。

## 五、Git 提交约定

1. 每次当前界面完成并确认后，提交一次 Git。
2. 提交信息应清楚描述本次完成内容，例如：
   - `完成启动页与创建人物页美术更新`
   - `完成家庭背景页功能版`
   - `完成家庭背景页正式美术替换`
3. 若当前阶段同时包含“流程文档更新”，提交信息中应体现。

## 六、当前已确认采用的流程

从当前版本开始，默认采用以下开发顺序：

1. 启动页、创建人物页已作为第一批样例界面。
2. 后续界面继续沿用“先功能临时版，后正式美术替换”的开发模式。
3. 每个界面完成并确认后，必须提交到 Git，再继续下一个界面。
